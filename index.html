在前三篇文章中，我们依次追踪了 LightRAG 查询流程中的关键词提取、批嵌入与向量检索、Graph Store 层编排。当检索阶段（Stage 1）完成，我们已经拿到了原始实体、关系和文本块数据。但直接把这些原始数据全部塞进 LLM 是有问题的：

  1. Token 预算有限：LLM 有 context window 限制（通常是 32K/128K tokens），而检索到的实体、关系和文本块加起来可能远超这个数
  2. 信息冗余：实体描述、关系描述、文本块之间存在大量重叠
  3. 格式要求：LLM 需要结构化的上下文，而不是原始 JSON

LightRAG 通过一个4 阶段的 Context 构建管线来解决这些问题：

本文深入 _apply_token_truncation、_merge_all_chunks、_build_context_str 三个函数，以及它们调用的 truncate_list_by_token_size、process_chunks_unified、convert_to_user_format 等工具函数，完整还原 Token 预算的计算与 Context 的最终构建。
