一、批嵌入（Batch Embedding）：为什么要一起嵌入？

通常的做法是：需要嵌入时就调一次 embedding API。但在 LightRAG 的 query 流程中，有三个文本需要嵌入：

需要嵌入的文本 用途 哪个 mode 需要
query 原文 搜 chunks VDB（当 kg_chunk_pick_method == "VECTOR" 或 mode == "mix"） local/global/hybrid/mix
ll_keywords（字符串拼接） 搜 entities VDB local/hybrid/mix
hl_keywords（字符串拼接） 搜 relationships VDB global/hybrid/mix

2.4 上下文感知嵌入（Asymmetric Embedding）

LightRAG 在 query 时传入 context="query"，在索引（插入）时可能传入 context="document"。这是为了支持不对称嵌入——部分 embedding 模型（如 Cohere Embed v3）对 query 和 document 使用不同的编码前缀：

supports_asymmetric 默认自动检测——如果包装的函数签名包含 context 参数，则自动启用。这个设计使得同一个 embedding 函数接口既能兼容对称模型，也能适配不对称模型，无需用户显式配置。

某些模型（如 OpenAI text-embedding-3-*）支持 API 缩短输出维度。当 send_dimensions=True 时，维度和配置会一并传给底层函数。

五、WFET 两种 Chunk 选择算法

六、实体和关系的 Round-Robin 去重合并

当 mode 是 Hybrid 或 Mix 时，local 和 global 路径各自产出了实体和关系列表。合并时采用 round-robin 交替合并 + 去重：

为什么用 round-robin 而不是先 local 再 global？

因为两个列表内部都是按重要性排序的（local 按 cosine，global 按 weight）。如果先全放 local 再 global，前 top_k 个全是 local 实体。round-robin 交替排列保证不同类型的重要实体在最终的 top N 中都有代表。

去重基于 entity_name 或 (src, tgt) 元组。同一个实体被两个路径都命中时只保留第一次出现的位置。

九、设计要点总结

  1. 一次嵌入原则： 尽量在一次 API 调用中完成所有需要嵌入的文本，减少 network round-trip。当某个 mode 不需要某条路径时，对应文本不嵌入。

  2. VDB + 图存储互补： VDB 负责”语义寻找”（哪些实体/关系跟关键词语义相关），图存储负责”拓扑补充”（这些实体有哪些属性、关联了哪些边、在图中的重要性）。两者通过 entity_name、(src, tgt) 键关联。

  3. WEIGHT 作为降级底线： VECTOR 模式依赖 embedding API + chunks VDB，这两者任一不可用时就回退到纯图权重。图拓扑本身就携带了足够的优先级信息。

  4. Round-robin 去重： 不是为了最大召回，而是为了多样性——local 和 global 来源的实体在最终列表中交替出现，避免某一类主导前几名的位置。

  5. 不对称嵌入的前瞻： 虽然当前默认 embedding 函数大多是对称的，但 EmbeddingFunc 的 context 参数为不对称 embedding 预留了完整的路由——query 和 document 使用不同的编码前缀，这在需要更精确的 query-doc 匹配时可以直接启用。

下一步：向量检索返回的实体和关系被转换成 token 字符串后，需要经过Token 预算与裁剪系统的管控——即 1.5 Token 预算与裁剪。
