LightRAG 源码分析 · 1.1 从提问到回答——Query 全链路拆解

发表于 2026-07-12 分类于 LightRAG源码分析 本文字数： 3.1k 阅读时长 ≈ 11 分钟
LightRAG 中用户调用 aquery("问题") 之后，代码一步一步去了哪里、每步做了什么、产生了什么数据、最终怎么变成 LLM 看到的 prompt。

章节定位： 这是 LightRAG 源码分析系列的第一篇。本文聚焦 Query 路径——从 aquery 入口开始，完整追踪所有 5 种查询模式（local / global / hybrid / mix / naive）的数据流与调用链。不涉及 Chunk ID 生成与 source_id 指针设计。

三、关键词提取

所有 KG 类模式的第一步，在 operate.py:3794 的 kg_query 中：

get_keywords_from_query（第 4009 行）逻辑很简单：

为什么不直接用 query 嵌入去搜？

这是 LightRAG 区别于 naive RAG 的核心设计。KG 检索需要”概念级”的匹配——不是找跟 query 字面相似的句子，而是在知识图谱中找到与 query 语义相关的实体和关系。LLM 抽出来的关键词是 query 中涉及的概念，用概念去搜实体 VDB / 关系 VDB，召回的是”同一个事物”而非”同一段话”。

抽取结果举例：

Query hl_keywords（高层） ll_keywords（低层）
“苹果公司今年的营收如何” “苹果, 科技公司, 财报” “苹果公司, 2024 年营收, 财务报表”

两个层级的后续用途完全不同：

  • ll_keywords → 搜 entities VDB（local 路径）
  • hl_keywords → 搜 relationships VDB（global 路径）

抽取结果会被 LLM 响应缓存命中时跳过（key 包含 hash 后的 query + 所有参数）。

4.1 Stage 1：Search——向量搜图 + 向量搜块

_perform_kg_search（第 4323 行）是本阶段执行函数。

批嵌入（batch embedding）

进入搜索逻辑之前，有一个关键的前置步骤：批预计算嵌入。根据当前 mode 判断本次需要哪些嵌入，一次调 LLM 嵌入接口算出所有，避免 2-3 次串行 API 往返。

搜索路由

根据 mode 不同有三种搜索方式：

条件 搜索行为
mode == "local" 只用 ll_keywords 搜 entities VDB → 取回实体 → 从图取关联关系
mode == "global" 只用 hl_keywords 搜 relationships VDB → 取回关系 → 从图取关联实体
mode in ("hybrid", "mix") 两者都做 + round-robin 合并去重 +（mix 专属）用 query 搜 chunks VDB

4.2 Stage 2：Truncation——Token 预算裁剪

搜回来的实体和关系数量可能远超出 LLM 窗口能容纳的范围，第二阶段做 token 预算裁剪。

_apply_token_truncation（第 4533 行）接受两个预算参数：

默认值分别是 6000 和 8000（可配置）。

裁剪策略： 按嵌入距离（搜索结果顺序）保留靠前的实体/关系，截断到预算内，保留原有顺序。裁剪后，从 entities_context 的 entity 字段反向查找原始实体数据，建立 filtered_entities 和 filtered_relations 供下一阶段用。

为什么保留顺序？因为排序本身是 embedding 相似度的体现——最相关的排在前面，被截掉的是相对不重要的。

4.3 Stage 3：Chunk 汇聚——从图到文本

实体和关系本身只有”概念名 + 描述 + 属性”，真正的文本内容在 chunks VDB 里。第三阶段做的核心操作是 reverse lookup：source_id → text_chunks。

_merge_all_chunks（第 4739 行）接受三个来源：

来源 数据 函数
vector_chunks query 向量搜 chunks VDB 直接在搜索阶段已返回
entity_chunks 实体的 source_id 对应块 _find_related_text_unit_from_entities
relation_chunks 关系的 source_id 对应块 _find_related_text_unit_from_relations

合并策略：round-robin 三路去重

去重逻辑： 同一个文本块可能被多个实体引用、也被关系引用、还被向量搜到。合并时第一次遇到加入，后面的跳过。三路交替取，保证每种来源的高分块都有机会进入最终集合。

如果启用了 enable_content_headings，合并后还会从 text_chunks_db 回填 heading path（文档中的章节路径），让 LLM 看到每段内容的上下文位置。

五、LLM 调用与缓存

context 全部组装好后，回到 kg_query（第 3941 行）准备最终调用 LLM。

第一步：cache 检查

Cache key 包含什么？ 不只是 query 本身，而是 (mode, query, response_type, top_k, chunk_top_k, max_entity_tokens, max_relation_tokens, max_total_tokens, hl_keywords, ll_keywords, user_prompt, enable_rerank, enable_content_headings, llm_identity) 的全维度 hash。只要这些参数中任何一个变化，cache 就不命中。

第二步：响应清洗

LLM 返回后，对非 streaming 响应做清洗：

有些 LLM 在返回中会携带输入 prompt 的回显，这里逐一剥离。streaming 模式不做此清洗（流式逐 token 吐，无法整体替换）。

返回的 QueryResult 包含 content（非流式响应）或 response_iterator（流式迭代器），以及组装好的完整 raw_data（供后续二次处理和数据分析）。

七、各模式的完整路径串讲

Local 模式（”搜特定实体”）

典型场景： “苹果公司的 CEO 是谁？”——需要精确实体识别。

Global 模式（”搜全局主题”）

典型场景： “2024 年科技行业的整体趋势如何？”——需要全局视角，不针对特定实体。

Hybrid 模式（local + global 合并）

典型场景： “苹果公司近期的营收受哪些宏观经济因素影响？”——既有具体实体又有全局背景。

Mix 模式（hybrid + chunks VDB）

Mix 与 Hybrid 的唯一区别：多了一步用 query 嵌入搜 chunks VDB，给下游提供额外”纯文本”视角。官方推荐搭配 reranker 使用。

Naive 模式（纯向量，不走图）

典型场景： 文档量小、不需要知识图谱的场景，或作为 ablative baseline。
