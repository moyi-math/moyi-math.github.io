RAG 学习笔记 · 1.3 RAG 基础概念与工作原理

摘要(Abstract)

TL;DR: RAG(Retrieval-Augmented Generation,检索增强生成)= 检索器 + 生成器。本质是把 1.2 的”参数记忆(LLM)”和”非参数记忆(外部知识库)”显式拼接:用 LLM 的”理解力”读外部资料,再用 LLM 的”推理力”写答案。Grounding(接地)机制能让幻觉率从 13% 压到 2%,但标准 RAG 自身也有 3 个固有缺陷——这正是 1.4 要讲的 Naive → Advanced → Modular 演进动力。

  • 标准 RAG 的 3 个固有缺陷:召回差、上下文污染、生成端幻觉,为 1.4 铺垫

适合读者:正在设计或评估企业级 AI 系统、需要判断 RAG 是否值得投入的工程师 / 架构师 / 技术决策者,以及准备 RAG 方向面试的候选人。

一、RAG 是什么?——一句话定义

RAG(Retrieval-Augmented Generation,检索增强生成) = 用一个外部检索器(Retriever)给 LLM “查资料”,再用检索到的内容增强(Enhance)LLM 的生成(Generation)结果。

把上面这句话拆成三段对应 1.2 的双记忆理论:

RAG 三段 1.2 中的对应角色 具体职责
Retrieval(检索) 非参数记忆 从外部知识库(向量库、搜索引擎、数据库)找到和问题最相关的文档片段
Augmented(增强) 双记忆的拼接 把检索到的片段 + 用户问题,组装成一个新的、带有参考资料的 prompt
Generation(生成) 参数记忆 LLM 基于这个增强后的 prompt 生成最终答案,并可附带引用

本质一句话:RAG 不是新模型,不是新算法,它是一种”把外部知识拼到 prompt 里”的工作流。LLM 本身可以不变,变的是你喂给它的 prompt 里多了”参考资料”这一段。

一个常见误区: 别把 RAG 当成”一种模型”。RAG 是 2020 年 Patrick Lewis 等人在 Facebook AI 提出的架构范式(见论文《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》),后续所有的 Advanced RAG / Agentic RAG / GraphRAG 都是这个范式的演进,没有改变 RAG = 检索 + 增强 + 生成 这个三段式骨架。

二、RAG vs 微调(Fine-tuning)——什么时候用哪个?

这是 2026 年 AI 工程师面试最高频的问题之一。企业 AI 项目立项时,第一个问题就是:”我该选 RAG 还是微调?”

1. 对比矩阵

维度 RAG(检索增强) 微调(Fine-tuning)
核心动作 在推理时检索外部知识拼到 prompt 在训练时改写模型参数
知识更新成本 几秒钟(往向量库里加文档) 几万到几百万美元(fine-tune 一次)
知识更新延迟 秒级(实时) 月级到年级(重训)
适合的知识 动态事实(产品文档、价格、新闻)、私有数据、长尾知识 稳定技能(特定语气、风格、输出格式)、领域术语理解
可解释性 ✅ 高(每条答案都能追溯到原始文档) ❌ 低(黑盒,难溯源)
幻觉控制 ✅ 强(有 grounding) ⚠️ 中(微调可降低部分幻觉,但难根除)
数据要求 中等(需要文档清洗 + 切块) 高(需要大量高质量问答对)
算力要求 低(只是检索 + 推理) 高(训练成本高)
适用场景 企业知识库、客服问答、文档助手 特定领域风格、代码助手、垂直行业语料

3. 2026 年的真实工程取舍

主流架构:RAG 为主 + 少量微调。

  • 80% 的企业 AI 系统走 RAG(因为知识随时在变)
  • 20% 的场景配合微调(例如客服话术统一、特定 schema 输出)

纯微调的应用场景已经很少了,因为:

  • 知识更新跟不上业务变化
  • 一旦微调完,模型就是”快照”,文档一改就要重训
  • 私有数据微调后塞不进 LLM 的参数里(压缩比 50:1,见 1.2)

一个反直觉的观察: 大模型越来越大(Long Context 1M+ tokens)后,有人以为”不需要 RAG 了,把全文档塞进去就行”。这是错的——

  • 1M tokens 也有极限,且成本高(Gemini 3.1 Pro 处理 1M tokens 完整 prompt 约 1.5/次)1.5/次)1.5/次)- Long Context 的 lost in the middle 问题(中间位置事实性错误率显著高于首尾)依然存在
  • RAG 把 100M 文档压缩成 top-5 片段,精确度 + 成本都更优

三、标准 RAG 的 5 步流水线

这是 1.3 的核心。RAG 工作流拆成两个阶段:离线(构建知识库) + **在线(回答问题)**。

2. 离线阶段详解

① 文档加载(Document Loading)

把各种格式的文档解析成纯文本。常用工具:

  • PDF: PyMuPDF、pdfplumber、Unstructured
  • Word: python-docx
  • HTML: BeautifulSoup、Trafilatura
  • Markdown: 直接读
  • Notion/Confluence: 各自 API

这一阶段常见问题:

  • PDF 中的表格、公式、扫描图片容易丢失结构
  • 多栏排版的 PDF 会被读成单栏
  • 需要 OCR 处理扫描件

② 文档切块(Document Chunking)

这是 RAG 工程质量的天花板之一。 切不好,再好的模型也白搭。

常见策略:

策略 块大小 优点 缺点
固定长度切块 200~512 tokens 简单、均匀 经常从句子中间切断,语义破碎
按句子/段落切 1~10 个句子 语义完整 块大小不均,检索粒度不可控
递归字符切块(Recursive) 自适应 平衡,LangChain 默认 切分点选择有经验成分
语义切块(Semantic) 变长 块内语义高度连贯 切分慢、需要 embedding 模型
按文档结构切 按 H1/H2/章节 结构保留好 仅适用于结构化文档

经验值:英文 200512 tokens / 中文 300800 tokens,后续 2.x 章节会专门讲切块的深度策略。

③ 向量化(Embedding)

把每块文本转成稠密向量,2026 年主流 Embedding 模型:

模型 维度 特点
OpenAI text-embedding-3-large 3072 综合最佳,生产稳定
BGE-M3 (BAAI) 1024 中英文都强,国产开源
M3E / BCE 768 中文场景性价比较高
Cohere embed-v3 1024 多语种、长文本
bge-large-zh-v1.5 1024 国产开源经典

关键:检索时,问题和文档必须用同一个 Embedding 模型——否则向量空间对不上,检索必失败。

④ 入库(Storage)

向量数据库选型:

类型 代表产品 适用场景
嵌入式 / 轻量 Chroma / FAISS 本地原型、小规模(<100万向量)
开源生产级 Milvus / Weaviate / Qdrant 大规模、可集群
云托管 Pinecone / Zilliz Cloud 不想运维
传统库加向量插件 pgvector / Elasticsearch 8+ 已有 PG/ES 栈

3. 在线阶段详解

① 检索(Retrieval)

常见检索模式:

  • 稠密向量检索(Dense Retrieval):用向量相似度(余弦、内积)找 top-K
  • 稀疏检索(Sparse Retrieval):BM25、TF-IDF,关键词匹配
  • 混合检索(Hybrid):两者结合 + Reranker 精排(2026 年最主流)

这个模板的三个要点:

  1. 明确”允许拒答”,而不是被迫胡编
  2. 强制带引用,提升可解释性
  3. 资料有序编号,LLM 容易对齐

③ 生成(Generation)与后处理

LLM 生成后,通常需要:

  • 提取答案 + 引用的链接
  • 加拒答检测(“我不确定”类回答高亮给用户)
  • 流式输出(SSE/WebSocket)
  • 评估 / 缓存等横切关注点

四、Grounding 机制 —— 为什么 RAG 能把幻觉率从 13% 压到 2%?

回到 1.1 的数据,对照看:

模型 无 RAG 幻觉率 加 RAG 后幻觉率 下降幅度
Claude Sonnet 4.6 ~6% ~2% -67%
GPT-5.4 ~7% ~3% -57%
Gemini 3.1 Pro ~13% ~5% -62%

1. 原理:让模型”看着资料写答案”

Grounding(接地):让 LLM 的回答有明确的、可追溯的、可验证的事实依据,而不是从参数分布里”猜”。

RAG 实现 grounding 的方式很直接:把检索到的真实文档片段塞进 prompt,让 LLM 在生成时被这些片段”锚定”。回答的内容必须能在这些片段里找到对应。

2. 为什么 grounding 这么有效?

根因:把 LLM 从”开卷猜测”变成”开卷作答”。

  • LLM 训练目标:最大化下一个 token 的概率
  • 在无 grounding 时,这个概率由参数里学到的模糊分布给出,自然会产生幻觉
  • 在有 grounding 时,prompt 里出现的事实片段强烈偏向于让模型复述/总结这些内容——概率分布被”压”到了真实片段上

这就像给学生两种考试:

  • 闭卷:全凭记忆 → 大概率会记错/编造
  • 开卷:可以翻书 → 答案准确率高很多,而且能引用页码

RAG = 给 LLM 开卷考试。

3. grounding 的副作用:over-reliance(过度依赖)

RAG 不是万能的。如果检索的片段本身就是错的、过时的、或问题与片段不相关,LLM 会努力从这些片段里”硬挤”出一个答案——这就是 **context-injected hallucination(上下文注入型幻觉)**。

这正是 1.4 要讲的”标准 RAG 缺陷”之一。

反直觉观察: 检索质量差的 RAG 系统,幻觉率可能反而高于纯 LLM。因为模型”勉强回答”会比”诚实拒答”更危险。

五、标准 RAG 自身的 3 个固有缺陷 —— 给 1.4 铺垫

重要:本节是 1.3 和 1.4 的桥。 标准 RAG(也叫 Naive RAG)虽然能把幻觉压下来,但工程上还有 3 个绕不开的问题。

1. 召回质量差(Retrieval Quality)

问题: 检索器找不到真正相关的文档。

  • 常见表现:用户问”A 产品保修期”,检索返回的是 B 产品的保修政策
  • 根因:Embedding 模型的语义捕捉能力有上限、Query 和文档不在同一向量空间(同义不同词)、文档切块导致上下文断裂
  • 表现指标:Recall@K(前 K 个结果里包含正确答案的比例)

2. 上下文污染(Context Contamination / Lost-in-the-Middle)

问题: 检索到了片段,但 LLM 没读到或读错了。

  • 常见表现:把片段拼进 prompt 后,LLM 忽略中间位置的事实(类似 Long Context 的 “lost in the middle”)
  • 解药:LongContextReorder(把重要片段放在首尾)、Reranker 精排、上下文裁剪

3. 上下文注入型幻觉(Context-Injected Hallucination)

问题: 检索到的内容本身有问题,LLM 配合”被迫回答”时反而编得更自信。

  • 常见表现:文档里有错别字 / 旧版本数据 / 模型”脑补”了文档之间的关联
  • 解药:文档清洗、版本控制、拒答检测、检索评估循环

1.3 的尾声:RAG 不是终点,是起点

正是因为这 3 个缺陷,2024-2026 年才演化出了 Advanced RAG、Modular RAG、GraphRAG、Agentic RAG——它们都在优化上面 3 个环节中的某几个。

七、面试快问快答(3 题覆盖 1.3 全部知识点)

Q1:RAG 和微调怎么选?如果两者都要,顺序是什么?

A: 见第二节决策矩阵。简答:

  • 知识更新频繁 → RAG
  • 改变输出风格/格式 → 微调
  • 两者都要 → 先微调风格,再加 RAG 喂知识(顺序不能反)

反着来(RAG 后接微调)的问题:微调会”覆盖”掉 RAG 喂的事实,反而把幻觉放大。

Q2:RAG 一定比 Long Context(百万级上下文)好吗?

A: 大多数场景下,RAG 优于 Long Context。 理由:

  • 成本:RAG 处理 top-5 片段 vs 1M tokens 输入,后者贵几十倍
  • 精度:Long Context 的 “lost in the middle” 问题,RAG 不存在(因为只挑相关片段)
  • 可解释:RAG 可精确到段,Long Context 答案无法溯源

Long Context 胜出的场景:需要跨多文档做整体推理(例如”比较三份合同的所有条款差异”)——这种任务 RAG top-K 检索容易漏信息。

Q3:RAG 的核心缺陷是什么?怎么解决?

A: 三大缺陷(见第五节):

  1. 召回差 → 改用混合检索 / Reranker / 微调 Embedding
  2. lost in the middle → LongContextReorder / 切片更短
  3. 上下文注入型幻觉 → 拒答检测 / 检索评估 / 文档清洗

进一步的高级范式(GraphRAG / Agentic RAG)在 1.4 详细讲。

八、本节核心要点

  1. RAG = Retrieval + Augmented + Generation:本质是”把外部知识拼进 prompt”的工作流,不是新模型。
  2. RAG vs 微调决策:动态/私有/事实性知识 → RAG;输出风格/格式 → 微调;实际项目大多两者结合。
  3. 标准 RAG 流水线:离线 4 步(加载→切块→向量化→入库)+ 在线 3 步(检索→Prompt 拼接→生成)。
  4. Grounding 的本质:让 LLM 从”凭记忆猜”变成”看着资料答”,幻觉率从 13% 压到 ~2%。
  5. RAG 不是银弹:标准 RAG 自身有 3 个固有缺陷——召回差、lost in the middle、上下文注入型幻觉。
  6. 工程经验:文档切块是 RAG 质量的天花板,Embedding 模型必须 query 和文档同款。
