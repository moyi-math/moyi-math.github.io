RAG 学习笔记 · 1.4 Naive RAG → Advanced RAG → Modular RAG 演进

发表于 2026-07-12 分类于 RAG学习笔记 本文字数： 5.4k 阅读时长 ≈ 20 分钟
三大 RAG 范式对比:Naive RAG(原始管道)、Advanced RAG(预/后检索优化)、Modular RAG(模块化重组)。每种范式具体优化了"召回差 / lost in the middle / 上下文注入幻觉"中的哪几个,2026 年的选型决策矩阵。

章节定位: 1.3 末尾留了三个钩子——召回质量差、lost in the middle、上下文注入型幻觉。本节把”标准 RAG 的 3 个固有缺陷”拆开,看 Naive、Advanced、Modular 三代 RAG 范式各自修补了哪几个缺陷。这也是面试官追问”你们 RAG 怎么做优化?”的标准答案。

摘要(Abstract)

TL;DR: RAG 的演进不是线性的”升级”,而是模块化堆叠——

  • Naive RAG(2020~2022):原始 5 步管道,3 个缺陷全中
  • Advanced RAG(2023~2024):在”检索前 / 检索后”插入 7 个优化模块,召回差和 lost in the middle 大幅缓解
  • Modular RAG(2024~2026):把任何模块拆出来重组,甚至接入 Agent 的工具循环,3 个缺陷都能针对性修补,且能跨任务、跨模态复用

每一代范式都对应清楚解决了 1.3 末尾的哪几个缺陷。2026 年的主流工程实践不”选某一种”,而是按场景从 Modular RAG 的模块池里挑需要的”积木”组合。

本文覆盖:

  • 三大范式对比矩阵:一张表看清 2020→2026 的 RAG 演进
  • Naive RAG 的全貌:为什么会失败、为什么它依然有价值
  • Advanced RAG 的 7 个核心模块:Pre-Retrieval 4 个、Post-Retrieval 3 个
  • Modular RAG 的设计哲学:从”管道”到”积木”
  • 2026 年选型决策:从 Naive 到 Modular,什么场景用什么
  • GraphRAG / Agentic RAG 在 Modular 框架下的位置:不是”另一种 RAG”,而是”另一种模块组合”
  • 面试高频题:4 道覆盖 Naive → Advanced → Modular 全部知识点的 RAG 范式面试题

适合读者:正在设计或评估企业级 AI 系统、需要判断 RAG 该做几层优化的工程师 / 架构师 / 技术决策者,以及准备 RAG 方向中高级面试的候选人。

一、回顾与提问:标准 RAG 为什么不够?

先回到 1.3 末尾的三个钩子。Naive RAG(也就是”标准 RAG 5 步流水线”)在 2026 年的企业生产里几乎没人直接用,因为它在三类高频场景下稳定失效:

缺陷 典型场景 失效表现
召回质量差 文档里写”GLM-5.2 2026.06.17 发布”,用户问”智谱最近发了什么模型?” 同义不同词,Embedding 检索召回 0 条
lost in the middle 检索到 5 条参考资料,用户在中间 LLM 答非所问,事实准确率比无 RAG 还差
上下文注入型幻觉 检索到的文档里有错别字/旧版本/无关片段 LLM “被迫回答”,胡说比无 RAG 更自信

这 3 个缺陷就是本节的主线——我们看每一代 RAG 范式具体修补了哪几个。

二、三大范式对比矩阵(一图看懂)

维度 Naive RAG (2020~2022) Advanced RAG (2023~2024) Modular RAG (2024~2026)
核心思想 固定 5 步管道 在”索引侧 + 检索侧”加优化模块 把任何模块拆出来重组,可堆叠可重排
架构形态 线性管道(单一路径) 管道 + 旁路优化(7 个常用模块) 模块化图(任意 DAG)
3 个缺陷修补情况 ❌ 全中 ✅ 召回差、⚠️ lost in the middle 部分缓解、❌ 上下文污染未解 ✅ 可针对 3 个缺陷都堆模块
检索前优化 无 Query Rewrite、HyDE、Query Expansion、Step-back Prompting 同左 + 可任意重写/Routing
检索后优化 无 Reranker、LongContextReorder、上下文压缩 同左 + Reflection、Cite-then-Answer
可组合性 低(写死管道) 中(可选开启某些模块) 高(模块可换可堆)
与 Agent 集成 不能 不能 能(Modular RAG 是 Agentic RAG 的子集)
实现复杂度 1 天 1~2 周 2~8 周(取决于模块堆叠)
适用场景 原型验证、教学演示、内部 demo 多数生产系统(Q&A、文档助手) 复杂问答、多跳推理、跨模态、企业级
代表论文 Lewis et al. 2020 原版 RAG 《Advanced RAG Techniques》,《Building Production RAG》 《Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks》, 2024
2026 年主流 ❌ 已被取代 ✅ 是多数生产系统骨架 ✅ 是复杂场景首选

一句话总结三代关系:

Naive RAG 是骨架,Advanced RAG 是加了肉的骨架,Modular RAG 是把肉都拆成可拼接的乐高积木。

一个常见误解: 把 Advanced / Modular 当成”比上一代更好的替代品”。不是替代,是叠加。Naive RAG 的 5 步管道在 Modular 框架下依然存在——只是每一步都可以被替换、重排、堆叠。

三、Naive RAG —— 一切的起点

1. 架构图

这就是 1.3 第三节的 5 步流水线,没有任何旁路优化。

2. 为什么 Naive 在生产里几乎一定失败?

用 1.3 末尾的三个缺陷逐一对:

(1)召回质量差 —— 没有 Query Rewrite

  • 用户问:”智谱最近新发布的旗舰模型是哪一款?”
  • 文档里写:”智谱 2026.06.17 发布 GLM-5.2”
  • Embedding 模型很可能召回不到——因为”旗舰模型”和”GLM-5.2”在向量空间里不近

(2)lost in the middle —— 没有 Reorder / 压缩

  • 检索回 10 条 chunk,后面 5 条用户没看到
  • LLM 也倾向于忽略中间位置的事实(2023 年 Liu 等人论文《Lost in the Middle》已证实)

(3)上下文注入型幻觉 —— 没有拒答检测和评估

  • 检索到一条”GLM-5.2 取消了联网功能”的过期文档
  • LLM 会”努力”拼凑成答案,反而比无 RAG 时更自信地回答错

3. Naive 还有哪些常见”工程问题”?

维度 Naive RAG 的具体痛点
切块 固定大小,经常切断语义
Embedding 单模态语义,数字/型号/缩写敏感
元数据 没时间戳、来源、版本
评估 无任何检索/生成质量指标

4. Naive RAG 什么时候仍然合适?

  • 原型验证 / Demo:半小时搭出来演示 RAG 是啥
  • 教学:讲清楚标准管道时用
  • 数据极少且问答极简单(< 100 文档,问题很直白)
  • 大模型本身上下文超长 + 文档全塞进去成本可接受 —— 极少场景

2026 年的判断: Naive RAG 几乎不直接生产。它最大的价值是作为参考骨架——让你能说清楚”我的 Modular RAG 用了 Naive 哪几个步骤”。

四、Advanced RAG —— 7 个优化模块详解

1. 总览图

Advanced RAG 在 Naive 的 5 步管道两侧插入了 7 个可选模块(其中 4 个在检索前、3 个在检索后):

注:有些”模块”是检索前(Pre-Retrieval),有些是检索后(Post-Retrieval),有些是索引侧。Advanced RAG 的核心创新是把这些可选项摆出来,让工程师按场景挑选。

2. 四个 Pre-Retrieval 模块(检索前)

模块 ①:Query Rewrite(查询改写)

解决问题:用户口语化提问 vs 文档正式表述,语义对不上。

做法:用一个 LLM 把用户的口语化问题改写成检索友好的表述。

原始 Query Rewrite 后
“智谱最近新发的旗舰是哪款?” “智谱 AI 公司 2026 年发布的旗舰大模型名称与发布日期”
“那个开源的工具咋用?” “LangChain 开源框架的安装和使用方法”
“上篇文章里我提到过的那个东西” “(结合上下文召回)用户在上一轮提到的 LLM 模型”

实现要点:

  • 用便宜快速的模型(GPT-4o-mini / GLM-4-Flash / DeepSeek-V3 Mini)
  • 通常 1~3 个改写结果合并检索
  • 多轮对话需要保留对话历史

模块 ②:Query Expansion / HyDE / Step-back(查询扩展)

解决问题:单一的 query 召回不全,需要”多角度打”。

三种做法对比:

做法 思路 例子
Query Expansion 加同义词、近义词、上下位词 “LLM” → “large language model, 大语言模型, GPT, Claude”
HyDE(Hypothetical Document Embeddings) 让 LLM 先猜一个答案,用”猜测答案”去检索 用户问”GLM-5.2 什么时候发布” → LLM 先瞎写一段”GLM-5.2 于 2026 年发布” → 用这段去检索
Step-back Prompting 先问一个”上位抽象问题”,再问原始问题 先问”GLM 系列有哪些版本?” → 再问”GLM-5.2 的发布时间”

2026 年的判断:HyDE 在小文档库里收益大,在百万级库里边际收益递减(因为 Embedding 已经够好)。Step-back 在多跳推理里非常有用(2.1 节会专门讲)。

模块 ③:Hybrid Retrieval(混合检索)

解决问题:Dense 向量检索找不到精确词、关键词(人名、型号、版本号)。

做法:BM25(稀疏) + Dense(密集)双检索,融合得分。

为什么需要 BM25:

  • 数字/版本号(“GLM-5.2”、”iPhone 17 Pro Max”)
  • 人名/专有名词(“Robert Downey Jr.”)
  • 缩写/术语(“RAG”、”ETL”)

这些往往和语义无关但字面精确,BM25 抓得很准。

2026 主流实现:Qdrant / Milvus / Elasticsearch 8+ 都原生支持混合检索。Cross-encoder Reranker 当作精排。

模块 ④:Reranker(精排)

解决问题:粗排(向量检索)挑出 top-50,但真正相关的不在 top-5,LLM 看错了。

做法:用Cross-Encoder 模型(慢但准)对粗排结果重新打分,只取 top-K(K=3~5)。

主流 Reranker:

  • bge-reranker-large(开源,国产)
  • Cohere Rerank 3.5(商业,质量稳定)
  • Jina Reranker(开源 + 商业)
  • bce-reranker-base_v1(国产开源)

性能经验:Cross-Encoder 比 Bi-Encoder(Embedding)慢 10100 倍,但**精度高 2040%**。所以是”先向量粗排 + 再 cross-encoder 精排”两阶段。

3. 三个 Post-Retrieval 模块(检索后)

模块 ⑤:LongContextReorder(长上下文重排)

解决问题:lost in the middle —— LLM 忽略中间位置的事实。

做法:把最相关的片段放到 prompt 的首尾,最不相关的放中间。

经验值:N → N/2 + N/2 反转(N 是检索数)。

模块 ⑥:Contextual Compression(上下文压缩)

解决问题:检索到的 chunk 里有冗余、与问题不相关、超长。

三种压缩:

  • 抽取式压缩:用 Embedding 计算 chunk 和 query 的相似度,过滤低分 chunk
  • 生成式压缩:用 LLM 把 chunk 改写成更短的、针对 query 的精简版
  • 句子级过滤:只保留与 query 相关的句子

适用场景:检索结果噪声大时;窗口预算紧时。

模块 ⑦:Refined Generation(精炼生成) / Cite-then-Answer

解决问题:LLM “看到资料但忘记引用”或”幻觉拼接”。

做法:在 prompt 里强制模型先列引用、再写答案:

**这是 1.3 第四节 Grounding 的”工程化落地”**。

4. Advanced RAG 修补了哪几个缺陷?

缺陷 Advanced RAG 怎么修 修补程度
召回质量差 ①Query Rewrite + ②Query Expansion + ③Hybrid + ④Reranker ✅ 大幅缓解(召回率 +30~60%)
lost in the middle ⑤LongContextReorder + ⑥Contextual Compression ⚠️ 部分缓解(压缩后明显,但没根治)
上下文注入型幻觉 ⑦Cite-then-Answer 强制引用,LLM 难再”凭空编” ⚠️ 部分缓解(文档本身就错时仍幻觉)

结论:Advanced RAG 在”召回差”上几乎根治,后两个缺陷留到 Modular RAG 修补。

五、Modular RAG —— 把一切做成积木

1. 设计哲学:从”管道”到”DAG”

Naive / Advanced RAG 都是线性管道(检索前 → 检索 → 检索后 → 生成)。
Modular RAG 是有向无环图(DAG):任何模块可被替换、跳过、循环、重排。

注意:这些”模块”不是新发明的,而是把 Advanced RAG 的 7 个模块、Naive 的 5 步都拆成可拼接的 LEGO 积木。

2. 核心模块分类(2026 主流 Modular RAG 框架的通用抽象)

模块类别 典型模块 作用
预处理 Query Rewrite、Query Expansion、Step-back 把用户问的”自然语言”翻译成”检索友好”
路由(Routing) Query Router、Source Selector 根据问题类型选走哪个检索器(本地 / Web / SQL)
检索 BM25、Dense、Hybrid、Web Search、SQL Query 多种检索方式并存
精排 Reranker、Cross-Encoder 二次过滤
后处理 LongContextReorder、Compression、Citation Extraction 整理送给 LLM 的内容
反思 / 验证 Reflection、Self-Critique、Sufficiency Check 检查检索结果是否够用——这是 Modular 相对 Advanced 的最大创新
生成 Answer、Cited Answer、Structured Output 最后的输出
记忆 Conversation Buffer、User Profile、Knowledge State 多轮对话上下文

3. Modular RAG 修补了哪几个缺陷?

缺陷 Modular RAG 怎么修 修补程度
召回质量差 Pre-Retrieval 模块自由组合 + 路由选最优检索器 ✅ 根治(组合效果远超单检索)
lost in the middle 后处理模块 + Reflection 主动缩短上下文 ✅ 根治
上下文注入型幻觉 Sufficiency Check + Reflection:检索结果不够时拒答或重新检索,不再”硬挤” ✅ 根治

关键洞察:Modular RAG 引入 Reflection(反思) 模块——这是它和 Advanced RAG 的本质区别。Advanced RAG 走”一条道走到黑”,Modular RAG 走”看情况回头补救”。

4. Reflection(反思)模块:Modular RAG 的灵魂

Reflection 让 RAG 系统具备”自查能力”:

实现方式:用一个小 LLM(便宜)当”反思器”,给 prompt 让它判断检索结果是否充分。常见的判断维度:

  • 检索文档是否覆盖问题的所有子问题?
  • 文档时效性是否足够?
  • 多文档之间是否有矛盾?

这就是 Modular RAG 修补”上下文注入型幻觉”的核心 —— 让系统在证据不足时主动拒答,而不是硬挤出答案。

5. Modular RAG 的工程实现(2026 主流)

框架 特点
LangChain LCEL 表达式化的模块链,Python 写管道,DAG 需要手动
LlamaIndex Workflow 原生 DAG,每个模块是 Step
DSPy 把 RAG 编译成可优化的程序,模块可被 LLM 自我调参
Haystack 2.x Pipeline as DAG,组件化最彻底
RAGFlow 国产开源,深度文档解析 + 模块化 RAG 一体

2026 年选型建议:

  • 轻量 + Demo:LangChain LCEL
  • 中等复杂 + 生产:LlamaIndex Workflow 或 Haystack 2.x
  • 极致优化 + 可学习:DSPy(模块由 LLM 自动选)
  • 国产深度文档:RAGFlow

六、GraphRAG 与 Agentic RAG 在 Modular 框架下的位置

1. GraphRAG —— 不是”另一种 RAG”,而是”模块池里加入了图检索模块”

GraphRAG(微软 2024 年提出):

  • 在 Modular RAG 的”检索模块”里多了一个图检索器
  • 知识库先转成知识图谱(实体-关系-实体三元组)
  • 检索时既查向量,又查图谱(查实体间多跳关系)

修补的缺陷:“多跳推理 + 全局问题” 召回差(传统 RAG 不擅长,需要图的关系链)。

典型场景:”我们公司的供应链有哪些单点故障?” —— 需要跨多个文档推理实体间关系,向量检索抓不到,图检索一查一个准。

2. Agentic RAG —— Modular RAG + Agent 循环

Agentic RAG:把 RAG 的检索/反思模块包进 Agent 的工具循环。

Agentic RAG = Modular RAG + Agent Tool Calling + Reflection Loop

修补的缺陷:所有 3 个 + 跨数据源 + 多步推理。但代价:慢(每步要 LLM 决策)+ 贵(多轮 LLM 调用)+ 难调试。

2026 主流场景:

  • 多跳问答:”A 公司的最大股东投资了哪些竞品?”
  • 跨数据源:”上个季度的销售数据中,华北区的退货率最高的 SKU 是什么?”
  • 工具组合(检索 + 写代码 + 跑 SQL)

3. 三大”高级范式”的本质总结

范式 本质 解决的问题
Advanced RAG Naive + 7 个优化模块 召回差 / 部分 lost in the middle
GraphRAG Modular + 图检索模块 多跳推理 / 全局性问题
Agentic RAG Modular + Agent 循环 + 多 tool 跨数据源 / 工具组合 / 复杂推理

三者不是并列关系,而是嵌套关系:

  • GraphRAG 是 Modular RAG 的一种配置
  • Agentic RAG 是 Modular RAG 的加上 Agent 调度

回到工程现实:你到底要选哪种?

2. 按团队能力选

团队规模 推荐范式

**不要问”选哪种 RAG 范式”,而要问”我的场景需要 Modular 里的哪几个模块”。这是 2026 年的 RAG 工程思维转变——从”范式选择”变成”模块组合”**。

八、面试快问快答(4 题覆盖全章)

Q1:Naive / Advanced / Modular RAG 的核心区别是什么?

A: 三句话——

  • Naive:固定 5 步管道,所有问题统一路径
  • Advanced:在管道前后插入 7 个优化模块(检索前 4 个 + 检索后 3 个),处理召回差和 lost in the middle
  • Modular:所有模块都是乐高积木,可拼可拆,引入 Reflection 让系统”自查”,处理上下文注入型幻觉

Q2:你负责的 RAG 系统怎么做优化?优化了哪几个模块?

A:(经典面试题)推荐回答模板:

  1. 先讲清楚当前用哪个范式(Advanced / Modular)
  2. 列出实际启用的模块(不是”理论可以用的所有模块”),如:
    • Pre-Retrieval:启用了 Query Rewrite + Hybrid Retrieval
    • Post-Retrieval:启用了 Reranker + LongContextReorder
  3. 说明每个模块修补了哪个具体缺陷(召回差 / lost in the middle / 上下文污染)
  4. 关键:量化效果——模块 A 开启后 Recall@K 从 X 提升到 Y,幻觉率从 Z% 降到 W%
  5. 接下来打算启用的模块(展示思考路径)

Q3:GraphRAG 是什么?什么场景用?

A:

  • 本质:Modular RAG 里加了一个图检索模块(知识图谱 + 图查询)
  • 解决什么:向量检索搞不定的多跳推理、全局性问题(例如”我们公司的供应链有哪些单点故障?”)
  • 代价:构建图谱成本高、文档结构变化时图谱要重算
  • 2026 场景:合规审查、复杂决策、跨实体关系问答

Q4:Agentic RAG 一定比 Advanced RAG 好吗?

A: 不一定。 Agentic RAG 的代价:

  • 慢:每步要 LLM 决策,延迟高 5~10 倍
  • 贵:多轮 LLM 调用,token 消耗指数级增长
  • 难调试:循环路径不可控,失败原因难定位

结论:能用简单 RAG 解决的,不要上 Agentic。Agentic 适合真正需要动态工具组合、跨数据源、多步推理的场景。

九、本节核心要点

  1. RAG 演进是模块化堆叠,不是替代:Naive 是骨架,Advanced 加肉,Modular 把肉拆成积木。
  2. 三个缺陷对照每代修复:
    • 召回差 → Advanced 大幅缓解,Modular 根治
    • lost in the middle → Advanced 部分缓解,Modular 根治
    • 上下文注入型幻觉 → Advanced 无法根本处理,Modular 通过 Reflection 模块根治
  3. Advanced RAG = Naive + 7 个模块(检索前 4 个 + 检索后 3 个):
    • 检索前:Query Rewrite、Query Expansion、Hybrid、Reranker
    • 检索后:LongContextReorder、Compression、Cite-then-Answer
  4. Modular RAG 的核心创新是 Reflection:让系统”自查”,证据不足时拒答或重检索,而不是硬挤出答案。
  5. GraphRAG / Agentic RAG 不是”另一种 RAG”:
    • GraphRAG = Modular + 图检索模块
    • Agentic RAG = Modular + Agent 循环
  6. 2026 选型思维:从”选哪种范式”变成”挑哪些模块组合”。
