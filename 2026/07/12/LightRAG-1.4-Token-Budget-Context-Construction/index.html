LightRAG 1.4：Token 预算与 Context 构建

发表于 2026-07-12 分类于 LightRAG 源码解读 本文字数： 3.3k 阅读时长 ≈ 12 分钟

在前三篇文章中，我们依次追踪了 LightRAG 查询流程中的关键词提取、批嵌入与向量检索、Graph Store 层编排。当检索阶段（Stage 1）完成，我们已经拿到了原始实体、关系和文本块数据。但直接把这些原始数据全部塞进 LLM 是有问题的：

  1. Token 预算有限：LLM 有 context window 限制（通常是 32K/128K tokens），而检索到的实体、关系和文本块加起来可能远超这个数
  2. 信息冗余：实体描述、关系描述、文本块之间存在大量重叠
  3. 格式要求：LLM 需要结构化的上下文，而不是原始 JSON

LightRAG 通过一个4 阶段的 Context 构建管线来解决这些问题：

本文深入 _apply_token_truncation、_merge_all_chunks、_build_context_str 三个函数，以及它们调用的 truncate_list_by_token_size、process_chunks_unified、convert_to_user_format 等工具函数，完整还原 Token 预算的计算与 Context 的最终构建。

1. Token 预算体系

1.1 QueryParam 中的 Token 字段

在 lightrag/base.py 中，QueryParam 定义了三个核心的 Token 预算字段：

三层预算体系：

  • max_entity_tokens（默认 6000）：实体描述部分最多保留的 token 数
  • max_relation_tokens（默认 8000）：关系描述部分最多保留的 token 数
  • max_total_tokens（可选）：最终 LLM 输入的总 token 预算（包含 system prompt、KG context、query、buffer 等全部开销）。当设置了 max_total_tokens，它实际上会覆盖所有子预算——_build_context_str 会用总的可用 token 减去各项固定开销后，将剩余 tokens 动态分配给文本块。

注意 max_token_limitation 是一个遗留的 typo 字段（拼写应为 limitation 而非 limitation），当前版本中建议使用 max_total_tokens。

1.2 Token 裁剪的基本工具：truncate_list_by_token_size

所有 Token 裁剪操作都基于同一个底层函数 truncate_list_by_token_size（lightrag/utils.py:2596）：

设计特点：

  • 贪婪截断：从头开始累加 token 数，一旦超过预算就截断
  • 按条目不可分割：要么保留整个条目，要么丢弃整个条目。不会在条目的中间截断
  • 线性顺序敏感：保留的是列表前面的条目。因此条目的排序顺序直接影响哪些信息被保留
  • Key 函数：通过 key 参数决定每个条目要编码的文本。在实体/关系上下文中，是整个 dict 的 JSON 序列化

为什么贪婪截断足够：因为实体和关系的排序已经通过 pick_by_weighted_polling/pick_by_vector_similarity 完成了重要性排序，所以从头开始截断就等价于保留最重要的条目。

3. Stage 3：Merge Chunks（_merge_all_chunks）

_merge_all_chunks（lightrag/operate.py:4739）负责将来自三个不同来源的文本块合并：

  1. vector_chunks：来自向量检索（Naive 模式）的文本块
  2. entity_chunks：从裁剪后的实体通过 _find_related_text_unit_from_entities 获取的文本块
  3. relation_chunks：从裁剪后的关系通过 _find_related_text_unit_from_relations 获取的文本块

3.1 Round-Robin 合并

Round-Robin 的设计意图：三种来源的文本块按轮流取一个的方式合并，保证每种来源的信息在前 N 个合并结果中都有分布。这样即使后续 token 截断丢弃了后半部分，三种来源的信息都仍有机会被保留。

去重机制：通过 seen_chunk_ids 集合做全量去重。同一个 chunk_id 不论来自哪个来源，只出现一次。这解决了实体和关系可能引用同一文本块的问题。

3.2 Content Headings 回填

在合并完成后，_merge_all_chunks 还会做一件重要的事情——回填 content_headings：

_attach_content_headings 通过 text_chunks_db.get_by_ids(chunk_ids) 从 KV 存储中查出每个 chunk 的原始数据，提取 heading 层级路径（如 "Introduction > Architecture > Storage Layer"），并截断到 DEFAULT_MAX_SECTION_CONTEXT_TOKENS（约 500 tokens）。这些标题路径会被计入后续的 token 预算中。

为什么需要回填：向量存储（chunks_vdb）中不包含 heading 字段，而实体/关系 chunk 在合并时也会丢失原先的 heading。回填让 LLM 能够理解每个文本块在原始文档中的位置层级。

4. Stage 4：Build Context（_build_context_str）

_build_context_str（lightrag/operate.py:4847）是整个 Context 构建管线的最后一步。它接收 Stage 2 的上下文数据和 Stage 3 的合并文本块，计算动态 Token 预算，最终产出 LLM 输入字符串和结构化的 raw_data。

计算公式意味着：实体和关系的 token 占用越多，文本块可用的 token 就越少。这形成了一个零和博弈——检索到的实体/关系越多，能保留的文本块就越少。

4.2 文本块统一处理管道

有了 available_chunk_tokens 后，调用 process_chunks_unified 对合并后的文本块进行处理：

process_chunks_unified（lightrag/utils.py:4335）包含 5 个处理步骤：

Step 1 - Rerank（可选）：如果启用了 reranker，调用 apply_rerank_if_enabled 对所有文本块进行语义重排序。reranker 根据文本块与查询的相关性打分，分数高的排在前面。

Step 2 - 最低 Rerank 分数过滤：如果启用了 rerank，按 global_config["min_rerank_score"]（默认 0.5）过滤低分文本块。

Step 3 - chunk_top_k 限制：如果设置了 chunk_top_k，只保留前 K 个文本块。

Step 4 - Token 截断：调用 truncate_list_by_token_size 按 available_chunk_tokens 预算截断。这是最终的 Token 预算落地。

Step 5 - 添加 ID：为每个保留的文本块添加 id 字段（DC1、DC2…），用于引用。

6. 总结

本文深入了 LightRAG 查询流程中的 Token 预算与 Context 构建机制：

阶段 函数 核心职责
Stage 1 _perform_kg_search 多源检索（已覆盖）
Stage 2 _apply_token_truncation 按 max_entity_tokens/max_relation_tokens 裁剪
Stage 3 _merge_all_chunks Round-robin 合并 + 去重
Stage 4 _build_context_str 动态预算分配 + 格式化输出

关键设计决策：

  1. 贪婪截断：用 truncate_list_by_token_size 做简单高效的顺序截断，依赖前序步骤已经完成重要性排序
  2. 零和预算：实体/关系占用越多 Token，文本块可用的越少，通过预计算实现精准分配
  3. Round-Robin 合并：保证多源文本块的公平分布
  4. 双输出：同时返回 LLM 字符串和结构化 JSON，兼顾推理和 API 调用
