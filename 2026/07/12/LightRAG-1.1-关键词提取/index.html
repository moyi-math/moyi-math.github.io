LightRAG 源码分析 · 1.1 关键词提取——从用户问题到图检索的第一跳

发表于 2026-07-12 分类于 LightRAG源码分析 本文字数： 3k 阅读时长 ≈ 11 分钟
用户提问后，LightRAG 如何用 LLM 提取 hl_keywords 和 ll_keywords？从 prompt 模板到 JSON 解析，从缓存设计到"为什么不直接用 query 嵌入"，一步步拆干净。

章节定位： 本系列聚焦 LightRAG 内部机制的逐层拆解。本文是 Query 方向的第一篇——关键词提取。接下来会陆续覆盖向量检索、图存储、Token 裁剪、Chunk 汇聚、LLM 调用与缓存等环节。

零、前置知识：为什么需要关键词？

在深入代码之前，先理解一个设计问题：LightRAG 为什么不用 query 嵌入直接搜图？

传统 naive RAG 的做法是：query → 嵌入 → 搜 chunks VDB（文本块的向量库）。但 LightRAG 的 KG 路径（local / global / hybrid / mix）走的不是这条路。它需要从知识图谱中找到与问题相关的实体和关系。

知识图谱中存储的是概念（实体名、关系描述），不是完整的句子。用 “苹果公司今年的营收如何” 这个完整句子去搜 entities VDB，嵌入距离上最接近的可能是 “我今天吃了苹果”（因为都有”苹果”和动词结构），而不是 实体: "苹果公司"。

解决方案：先用 LLM 从 query 中抽取出核心概念（关键词），用这些概念去搜图的 VDB。关键词越精准，图检索的命中率越高。

LightRAG 将关键词分为两层：

层级 缩写 含义 检索目标 示例
high-level hl_keywords 高层概念、主题领域 relationships VDB（关系库） "苹果, 科技公司, 财报"
low-level ll_keywords 具体实体、专有名词 entities VDB（实体库） "苹果公司, 2024 年营收, 财务报表"

接下来看代码如何实现这一设计。

一、入口：get_keywords_from_query

文件： operate.py:4009

逻辑极简——两条路径：

  1. 短路径： QueryParam 中已传入 hl_keywords / ll_keywords，直接返回，不调 LLM
  2. 长路径： 调 extract_keywords_only 让 LLM 抽取

短路径的设计意图：高级用户可以在 QueryParam 中预指定关键词，精确控制检索方向，绕过 LLM 抽取的不确定性。

2.3 Cache 检查

cache 命中的结果不能直接信任——要用 _parse_keywords_payload 重新解析验证。如果缓存数据损坏了（比如旧版本格式变更），会优雅降级到重新调用 LLM。

2.4 Prompt 组装

kw_prompt 是一个 12 条规则约束的 LLM prompt。完整内容在 prompt.py:484，核心结构如下：

看几个值得注意的约束：

  • 规则 5（短语优先）： “Machine Learning” 应该作为单个短语保留，而不是拆成 “Machine” 和 “Learning”。这是 prompt 级别的防劣化设计。
  • 规则 6（空查询容忍）： 对无意义 query 返回空列表，而不是胡乱编造关键词。这是 prompt 级别的安全边界。
  • 规则 8（语言控制）： 提取结果的语言与 query 语言解耦，可以配置。专有名词保留原文。
  • 规则 9（模板隔离）： 显式声明 ---Output Format Template--- 里的内容不是源文本。防止 LLM 把 "<high_level_keyword>" 这个占位符当关键词提取出来。

2.5 LLM 调用

两个关键点：

role 隔离： LightRAG 的 LLM 调用体系按 role 分配不同的 LLM 函数。关键词提取用的是 role_llm_funcs["keyword"]——可以配置独立的模型、端点和参数，与主回答 LLM 分离。这意味着你可以让关键词提取走更便宜的小模型，主回答走更强的模型。

结构化输出： 传入 response_format={"type": "json_object"}，要求 LLM 以 JSON 格式返回。这是 OpenAI-compatible 接口的标准做法，能大幅降 parse 失败率。但并非所有 LLM provider 都支持此参数——不支持的 provider 会在后续 JSON 解析阶段通过 json_repair 做容错恢复。

这是全系列容错最精细的解析函数之一，因为 LLM 的输出格式五花八门，必须层层兜底。

解析流程：

逐层看。

_strip_markdown_code_fence 处理：

有些 LLM 虽然在 prompt 中被告知”不要 code fence”，但偶尔还是输出 ```json。这里做静默修正，只打 debug 日志不让用户感知。

json_repair 兜底：

标准 JSON 解析失败后降级到 json_repair，这是一个专为 LLM 输出设计的容错 JSON 解析器，能处理末尾多余逗号、属性名缺引号、单引号代替双引号等常见问题。如果连 json_repair 都失败，才真正放弃——打 error 日志，返回空列表。

规范化：_normalize_keyword_list（第 4041 行）：

这个函数处理两种常见情况：

  • 字符串型输出： 有些 LLM 不按 JSON 约定走，把关键词以逗号分隔符拼成一个字符串。这里按 \n、,、; 三种分隔符拆分。
  • 脏数据： 空字符串、纯空白、None 都被过滤。非字符串元素（如数字、嵌套对象）被跳过并打 warning。

2.7 写回 cache

保存到 KV Store（默认是 JsonKVStorage，写入 kv_store_llm_response_cache.json）。注意写入条件：

  • 只在 enable_llm_cache = True 时写
  • 只在 hl_keywords or ll_keywords 非空时写（空结果不缓存，下次重新提取）
  • queryparam_dict 额外保存了 query 参数上下文，方便后面检索时分析命中了哪些参数组合

五、设计要点总结

  1. 关键词 ≠ 嵌入： 关键词提取是为了”概念级匹配”，和向量检索是互补而非替代关系。关键词命中知识图谱的节点/边，向量相似度决定排序。

  2. 两层分离： hl（高层→关系库）和 ll（低层→实体库）对应两种不同的检索粒度，后续会进入不同的路由分支。同一组 LLM 调用同时产出两者，没有额外开销。

  3. Cache 是全维度的： 不只是 query 文本，mode、language、LLM 模型身份都被纳入 cache key。切换任何一个维度都不命中旧缓存。

  4. 容错分层： json.loads → json_repair.loads → 空列表 三层降级，配合 _normalize_keyword_list 的字符串/列表兼容，覆盖了几乎所有主流 LLM provider 的输出格式变化。

  5. Role 隔离： 关键词提取使用独立的 role_llm_funcs["keyword"]，可以与主回答使用不同的模型配置。这是 LightRAG 整套 LLM 调用体系的设计基础，后续文章会深入展开。
