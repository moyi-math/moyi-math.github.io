LightRAG 源码分析 · 1.3 Graph Store 层编排——从 VDB 结果到图拓扑数据与文本块的全链路

发表于 2026-07-12 分类于 LightRAG源码分析 本文字数： 4k 阅读时长 ≈ 15 分钟
向量检索找到候选实体和关系后，Graph Store 层如何编排图拓扑查询、边扩散、文本块关联？NetworkX 实现细节、批量操作、跨进程同步、WEIGHT/VECTOR chunk 选择算法，一网打尽。

系列定位： 本文是 LightRAG Query 方向的第三篇。前两篇分别拆解了关键词提取（1.1）和批嵌入与向量检索（1.2）。本文承接其上，深入到向量检索之后——Graph Store 层的编排逻辑。

零、引入：向量检索之后发生了什么？

在 1.2 中，我们追踪到三路 VDB 查询返回了候选实体和关系的 ID。但 VDB 只存储了实体/关系的名称和向量，（默认的 NanoVectorDB）不会存实体的完整属性描述、不会存边的权重，更不会存图拓扑结构中的度数。

Graph Store 层就是补全这些缺失信息的地方——它从图存储（默认 NetworkX）中批量拉取拓扑数据，合并到 VDB 结果上，并进一步扩散到边/实体和文本块。

整个链路分为三个阶段：

2.3 为什么批量方法重要？

get_nodes_batch 和 get_node_edges_batch 等批方法在默认 NetworkX 下是逐个调用，在 Neo4j/PostgreSQL 后端下则会被重写为一条 Cypher/SQL 查全部。接口一致，性能不同——这是策略模式在存储层的典型应用。

3.2 读写路径分离

写入索引时： 在 index_done_callback（networkx_impl.py:694-753）中，持有写锁将整个 self._graph 序列化到 GraphML 文件，然后通过 shared_storage 设置 storage_updated 标志。

关键设计： 这是读多写少的发布-订阅模式。索引写入是低频操作（文档入库时一次），查询是高频操作。每次查询时检查原子标志位（无需锁），只在有更新时触发重载。

5.2 _find_most_related_edges_from_entities（边扩散）

这是从实体到关系的关键桥梁。

算法：

为什么用 tuple(sorted(e)) 去重？ 因为 networkx.Graph() 是无向图，graph.edges("A") 返回的边以 ("A", "B") 格式呈现，但 graph.edges("B") 会返回 ("B", "A")。两个实体各自的 get_node_edges 可能产生同一无向边的两个方向，必须归一化后去重。

为什么先处理 VDB 相似度高的实体？ node_names 的顺序是 VDB 结果的余弦相似度降序。实体被优先扫描，它的关联边就优先进入 all_edges 列表。排序时虽然按 (rank, weight)，但等 rank+weight 的条件下，高相似度实体的边排在前面。

5.3 _get_edge_data（global 路径入口）

文件： operate.py:5427

与 _get_node_data 对称，但方向相反：VDB 查到关系 → 补图属性 → 追溯到实体。

Global 模式和 Local 模式的关键区别：

维度 Local (_get_node_data) Global (_get_edge_data)
查询目标 entities VDB relationships VDB
扩散方向 实体 → 边 关系 → 实体
排序依据 实体：VDB 余弦；边：(rank, weight) 关系：VDB 余弦；实体：无 rank
拓扑重要性 节点度数参与排序 边的 rank 不重新计算

十、设计要点总结

  1. 三层存储各司其职： VDB 负责”找到谁”，Graph 负责”谁有什么属性、跟谁相连”，KV 负责”内容是什么”。每一层都有可替换的后端实现。

  2. 拓扑排序只出现在边扩散阶段： 实体和关系的首次排序完全基于 VDB 余弦。拓扑信号（rank）只在从实体扩散到边时介入——边的排序是 (rank, weight)。这保证了”语义匹配”优于”拓扑重要”。

  3. Global 路径更扁平： Global 路径不计算 rank，关系排序保持 VDB 原始顺序。这是合理的——high-level 关键词匹配到的本应是全局主题，不依赖局部拓扑。

  4. WEIGHT 是默认且可靠的： 加权轮询不依赖任何外部服务（无需 VDB、无需 embedding 函数），仅靠图拓扑就能工作。VECTOR 是可选增强，失败时自动降级。

  5. 逐节点边的代价： 默认 NetworkX 的 get_nodes_edges_batch 是 O(候选数 × 平均度数)，通过并行 IO（asyncio.gather）隐藏延迟但无法降低计算量。高连通图建议切换到 Neo4j 等原生后端。

  6. 跨进程同步是乐观的： 不阻塞读取、写入后发信号、读取时惰性重载。适合索引频率低、查询频率高的场景。
