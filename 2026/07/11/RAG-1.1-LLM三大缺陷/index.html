RAG 学习笔记 · 1.1 LLM 三大缺陷

发表于 2026-07-11 分类于 RAG学习笔记 本文字数： 3.1k 阅读时长 ≈ 11 分钟
RAG 在 2026 年仍是企业级 AI 的基础能力层，不是补救措施。

2026 年的开场白: 2013 年 DeepMind 让 AI 学会打游戏,2023 年 ChatGPT 让 AI 学会聊天,2026 年的 AI 已经能连续自主工作 8 小时交付完整工程——但“能力越强 ≠ 越可靠”。LLM 三大缺陷仍然在,而且因为能力变强,反而更容易”自信地胡说”。

摘要(Abstract)

TL;DR: LLM 三大缺陷——幻觉 / 知识截止 / 上下文限制——是 RAG(检索增强生成)存在的根本理由。本文系统拆解三大缺陷的成因、2026 年实测数据,以及 RAG 在企业 AI 系统中从”补救措施”升级为”基础能力层”的必然性。

本文覆盖:

  • 三大缺陷的统一视角:为什么三大问题本质同源——模型是”文字接龙机器”,没有”查资料”动作
  • 2026 年真实数据:Claude Sonnet 4.6、GPT-5.4、GLM-5.2、Gemini 3.1 Pro、DeepSeek V4 等主流模型的幻觉率、价格、上下文真实可用性
  • RAG 新范式:GraphRAG、Agentic RAG、Memory-Augmented AI 如何应对三大缺陷
  • 工程实践:幻觉治理的多层防线、LongContextReorder 等可落地技巧
  • 面试高频题:5 道覆盖度完整的 RAG 面试题及参考答案

适合读者:正在设计或评估企业级 AI 系统、需要判断 RAG 是否值得投入的工程师 / 架构师 / 技术决策者,以及准备 RAG 方向面试的候选人。

一、幻觉(Hallucination)

1.1 真实案例:2026 年的”翻车现场”

Claude 4 在长会话里”自信地编代码”
Claude Code 在长会话中经常会引用不存在的 API endpoint,编造函数签名——这是 Claude 4 系列在 Agent 场景的”著名 bug”。而 Gemini 3.1 Pro 一次性输入百万 token 时,中间位置的事实性错误率显著高于首尾。

1.2 为什么再先进的模型都根除不了幻觉?

LLM 干的事情从头到尾只有一件:给定上文,预测下一个 token 最可能是什么。

它不是知识库,不是搜索引擎,不是数据库——它是一个超级强大的”文字接龙”机器。这个本质决定了三个根因:

  • 训练数据的噪声和矛盾:互联网 A 说 1+1=2,B 说 1+1=3,C 说 1+1 看情况,模型学到的是概率分布,不是事实
  • 概率采样的随机性:解码时(尤其是 temperature > 0)有随机性,可能在几个候选里选到不准确的那个
  • 缺乏 grounding:模型没有”查资料”这个动作,纯粹靠参数化的”记忆”在生成,细节错了它自己也不知道

1.3 2026 年真实幻觉率排行

模型 综合幻觉率 上下文幻觉率 亮点 / 备注
Claude Sonnet 4.6 ~6% ~2% 上下文忠实性最佳
GPT-5.4 ~7% ~3% 综合最优,生产稳定
GLM-5.2 ~9% ~3% AA 幻觉评测 86 分(国产第一梯队)
Gemini 3.1 Pro ~13% ~5% 长文本衰减严重,幻觉偏高

反直觉观察:Gemini 3.1 Pro 的”高幻觉率”

2026 年一个反直觉的现象:Gemini 3.1 Pro 的幻觉率反而比 Claude 和 GPT 高。原因:Gemini 百万级上下文带来的”长上下文失真”问题更突出,信息召回率在 128K 以上断崖式下跌。

1.4 2026 年幻觉的六大类型

类型 表现 2026 案例
事实性幻觉 编造不存在的实体、事件、数字 GLM-5.2 在 Rust / 海外新库上偶发
引用幻觉 编造论文、链接、来源 o3、DeepSeek-R1 arXiv ID 编造率 > 5%
数字幻觉 数字张冠李戴或多算少算 所有 SOTA 模型仍有
逻辑幻觉 推理链在某一步断裂 复杂数学题做错但过程流畅
Agent 代码幻觉 调用工具时编造参数 Claude Code 编造 API endpoint
长上下文失真 百万上下文中间段错误率激增 Gemini 3.1 Pro 128K 以上

2.3 知识截止的两个维度

公开知识 vs 私有数据

  • 公开知识的截止:互联网上的公开数据,模型没见过。可以通过联网搜索(RAG with Web Search)解决
  • 私有数据的缺失:企业内部文档、个人笔记、专业领域的资料——模型压根就没见过。后者才是企业级应用的真正痛点

2.4 为什么截止?—— 训练成本的物理限制

训练一个 SOTA 模型的代价:

  • OpenAI 估值 8520 亿美元(2025 年融资后)
  • Anthropic、Google、Meta 每年烧数十亿到上百亿美元
  • 几万张 GPU 跑几个月
  • 几 TB 到几十 TB 的清洗后数据

不可能每周重新训一次。截止时间是大模型的物理约束,不是工程 bug。

2.5 2026 年新趋势:开源 = 知识更新权

GLM-5.2 MIT 开源的战略意义

海外模型的知识更新受限于厂商训练节奏,国产开源模型(GLM-5.2、DeepSeek V4)让用户掌握更新权——本地微调、热更新、领域微调,不再受制于厂商。这是 2026 年 RAG 系统本地化、私有化、定制化的基础。

GLM-5.2:2026 年 1M 真可用的代表

GLM-5.2 完成了 88 万 tokens 的多端应用(Web/移动端/小程序)开发,几乎用满 1M 窗口。从需求分析、架构设计、代码编写、联调测试到打包上线,一次跑完。通过 IndexShare 机制让 1M 推理成本可控(单位 token FLOPs 降低 2.9 倍)。

3.2 子问题 1:Lost in the Middle(中间迷失)—— 2026 年更严重

Liu et al. 2023 的经典论文发现:长上下文中部信息注意力弱。

2026 年的实测数据:

  • Gemini 3.1 Pro 百万级上下文存在严重衰减,128K 以上信息召回率断崖式下跌
  • Claude Opus 4.8 在 1M 上下文中部也有类似问题
  • 2026 年企业实践中,100K 以上的上下文基本不直接塞 prompt,而是先做 RAG 预处理

3.4 子问题 3:延迟与子问题 4:位置偏差

延迟:1M token 仍然需要 5-10 秒 Prefill。RAG 方案通常 prompt 长度控制在 4K-16K,Prefill 时间毫秒级。

位置偏差:开头权重最高,结尾次高,**中段 50% 位置召回率比首尾低 30-50%**。RAG 检索结果的顺序很重要——最相关的放最前面。

2026 面试高频题

Q1:为什么 RAG 重要?LLM 三大缺陷是什么?

LLM 三大缺陷:幻觉(GPT-5.4 ~7%、Gemini 3.1 Pro ~13%)、知识截止(2025 年底)、上下文限制(Gemini 2M 但 128K 衰减)。RAG 通过外部知识注入同时解决三个问题。

Q2:Gemini 3.1 Pro 标称 2M 上下文,实际能用吗?

不能。128K 以上信息召回率断崖式下跌,中间位置事实错误率显著高于首尾。Lost in the Middle 在百万级上下文仍存在。GLM-5.2 的 1M 是 2026 年少数”真可用”的代表,通过 IndexShare 架构降低推理成本,实战处理 88 万 tokens。

Q3:为什么 RAG 在 2026 年反而更重要?

四大原因:①企业私有数据 100% 缺失的痛点没变;②模型越强、越敢”自信地胡说”,grounding 价值越高;③Agent 时代需要 RAG 提供可追溯的决策依据;④RAG 是 SOTA 模型的基础能力层,不是补救。2026 年新形态:Agentic RAG、GraphRAG、Memory-Augmented AI,Naive RAG 正在被同质化。

Q4:国产模型(GLM-5.2、DeepSeek V4)在 RAG 场景下有什么特别?

三大独特优势:①用户掌握知识更新权(MIT 协议开源,本地微调);②国内直连顺畅(无国际网络);③性价比碾压(DeepSeek V4-Flash $0.14/M vs Claude Opus 4.8 $5/M,差 35 倍)。痛点:对海外最新库/新文档覆盖偏弱,RAG 更刚需。

Q5:2026 年 RAG 有什么新形态?新机会在哪?

四大新范式:①GraphRAG(从向量相似度到知识图谱);②Agentic RAG(检索成为行动循环);③长期记忆系统(Memory-Augmented AI);④无检索推理(部分场景被更高架构吸收)。新机会:GraphRAG 工具化、Agent 记忆框架、低成本私有部署——单纯的 Naive RAG 正在被同质化。

结论(Conclusion)

三大缺陷不会消失,只会随着模型规模变大变得更隐蔽、更自信。RAG 在 2026 年从”补救 LLM 缺陷的工具”演变为**企业 AI 系统的”基础能力层”**——没有 RAG,任何严肃的 AI 应用都无法保证事实性、时效性和可追溯性。

工程实践上:

  • 不要试图消除幻觉,而是建立多层防线(预防 + 检测 + 监控)
  • 不要盲信上下文窗口大小,128K 是 2026 年的实际可用边界
  • 不要把 RAG 当成单次流程,而是设计为 Agent 循环的一部分(Agentic RAG)
